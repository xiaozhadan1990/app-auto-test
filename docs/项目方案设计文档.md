# 移动自动化测试桌面平台方案设计文档

## 1. 文档概述

### 1.1 文档目的

本文档用于描述 `app-auto-test` 项目的整体方案设计，沉淀当前系统的架构思路、模块职责、核心流程、数据设计与后续演进方向，作为后续开发、维护、扩展和评审的统一参考。

### 1.2 适用范围

当前方案覆盖以下系统组成：

- Python Flask 后端服务
- React + TypeScript 桌面 Web 前端
- Pytest + Appium 自动化执行链路
- SQLite 运行态数据存储
- HTML/JSON 测试报告管理
- 可选的 WebSocket 远程控制能力

### 1.3 设计原则

- 入口轻量：启动脚本只负责初始化与装配，不承载业务逻辑
- 服务分层：API 层保持薄，核心逻辑下沉到 service
- 状态一致：内存、数据库、前端轮询状态保持一致
- 路径兼容：同时支持源码运行与 PyInstaller 打包运行
- 单设备互斥：同一设备同一时刻只允许一个任务执行
- 渐进扩展：先满足 Android，保留 iOS 接入空间

## 2. 项目背景与目标

### 2.1 项目背景

本项目并非单纯的测试脚本仓库，而是一个“测试执行平台”。它面向移动端自动化测试场景，既需要组织 Appium/Pytest 用例执行，又需要提供图形化界面完成设备查看、任务触发、执行跟踪、历史查询与报告展示。

### 2.2 建设目标

- 提供桌面端一体化测试操作台，降低测试执行门槛
- 支持多应用、多测试包、多测试范围的灵活触发
- 自动记录任务历史、设备状态和报告摘要
- 支持截图、视频等测试附件的统一访问
- 支持本地运行与远程接入两种模式
- 为未来 iOS、调度、权限、分布式执行等能力预留扩展点

## 3. 总体架构设计

### 3.1 架构概览

系统采用前后端一体化的桌面服务架构：

1. 桌面启动入口加载环境变量、路径配置和服务容器
2. Flask 提供本地 HTTP API 与静态页面托管
3. React 前端通过 API 获取设备、任务、报告等数据
4. 后端通过 `adb` 获取设备信息，通过 Appium + Pytest 执行测试
5. 执行结果写入文件与 SQLite，再反馈给前端展示
6. 可选通过 WebSocket 接收远程命令并回传状态

### 3.2 分层结构

#### 表现层

- `web-ui/src/App.tsx`
- `ui/` 构建后的静态资源

职责：

- 设备列表展示
- 测试任务配置与触发
- 执行日志与任务历史展示
- 测试报告摘要与明细展示

#### 接口层

- `desktop_app/api.py`

职责：

- 定义 Flask 路由
- 解析参数
- 调用服务容器
- 返回 JSON / 文件响应

#### 服务编排层

- `desktop_app/services_container.py`

职责：

- 统一装配后端能力
- 管理依赖注入
- 作为 API 与具体服务之间的协调中心

#### 领域服务层

- `desktop_app/task_service.py`
- `desktop_app/device_service.py`
- `desktop_app/report_service.py`
- `desktop_app/db_service.py`
- `desktop_app/package_service.py`
- `desktop_app/remote_ws_service.py`

#### 基础设施层

- Appium Server
- ADB
- Pytest
- SQLite
- 本地文件系统
- 可选远程 WebSocket 服务

## 4. 核心模块设计

### 4.1 启动入口模块

文件：`desktop_web_app.py`

职责：

- 识别源码模式与打包模式
- 解析 `RESOURCE_ROOT` / `RUNTIME_ROOT`
- 加载本地 `.env`
- 创建 `DesktopServiceContainer`
- 初始化运行态数据库
- 按需启动远程 WebSocket 客户端
- 启动 Flask/Waitress 服务并自动打开浏览器

设计说明：

- 启动入口仅承担初始化与托管职责，避免业务逻辑膨胀
- 所有业务能力通过服务容器注入到 Flask 应用

### 4.2 服务容器模块

文件：`desktop_app/services_container.py`

职责：

- 统一封装任务、设备、报告、数据库、远程控制能力
- 维护任务运行时对象 `TaskRuntime`
- 对外暴露 API 依赖集合 `ApiDeps`
- 维护启动探针与 Appium 就绪缓存

设计说明：

- 作为后端组合根，降低 API 与底层服务的耦合
- 统一解决路径、运行态对象、环境变量解析等横切问题

### 4.3 设备管理模块

文件：`desktop_app/device_service.py`

职责：

- 调用 `adb devices` 获取在线设备
- 通过 `getprop` 获取品牌、型号、系统版本
- 通过 `dumpsys package` 获取目标应用版本
- 使用短时缓存降低重复探测开销

设计说明：

- 当前以 Android 为主，天然依赖 `adb`
- 设备信息与设备运行态分离：静态信息来自 ADB，运行态来自 SQLite

### 4.4 测试包管理模块

文件：`desktop_app/package_service.py`

职责：

- 根据 `app_key` 返回可执行测试包
- 扫描 `tests/<app>/test_*.py`
- 尝试从 `@pytest.mark.case_name(...)` 提取中文展示名

设计说明：

- 兼顾“目录级执行”和“单文件执行”
- 前端展示名与实际执行路径解耦

### 4.5 测试执行模块

文件：`desktop_app/task_service.py`

职责：

- 校验运行参数与 Appium 状态
- 生成 pytest 命令
- 为任务创建独立日志与报告目录
- 启动 pytest 子进程
- 维护内存中的任务状态与设备占用关系
- 执行结束后处理报告与数据库落库

关键约束：

- 同一设备不可并发执行多个任务
- 执行过程必须同步更新：
  - 内存任务状态
  - SQLite 任务历史
  - SQLite 设备状态

设计说明：

- 任务启动与任务收尾通过 watcher 线程异步完成
- 前端通过轮询 `/api/task_status/<task_id>` 感知进度

### 4.6 报告管理模块

文件：`desktop_app/report_service.py`

职责：

- 计算任务级报告路径
- 判断 HTML 报告与结构化报告数据是否存在
- 读取 `test_results.json` 并落库摘要与明细
- 生成附件访问 URL
- 在远程模式下上传截图/视频并改写资源地址

设计说明：

- 采用“文件落地 + 数据库存储摘要”的双轨模式
- 既保留原始 HTML 报告，又支持前端结构化展示

### 4.7 数据持久化模块

文件：`desktop_app/db_service.py`

职责：

- 初始化 SQLite 表结构
- 维护设备运行态
- 维护任务历史
- 维护报告摘要与报告明细

表设计：

- `device_runtime_status`
  - 存储设备当前状态、关联任务、提示信息、更新时间
- `task_run_history`
  - 存储任务维度执行记录、退出码、日志路径、报告处理结果
- `task_report_summary`
  - 存储任务级统计信息
- `task_report_cases`
  - 存储用例级明细、截图、视频、错误信息

### 4.8 远程控制模块

文件：`desktop_app/remote_ws_service.py`

职责：

- 按环境配置决定是否启用 WebSocket 客户端
- 与远端服务建立连接、注册、心跳、重连
- 接收远程命令并转发到本地服务容器
- 记录远程日志到 `reports/remote-ws.log`

设计说明：

- 远端仅通过标准 action 调用本地既有能力
- 本地 HTTP 服务仍是主控制面，WebSocket 是补充通道

## 5. 前端方案设计

### 5.1 技术选型

- React 18
- TypeScript
- Vite
- Ant Design

### 5.2 页面能力

前端当前主要集中在 `web-ui/src/App.tsx`，包含四个核心页面域：

- 手机终端
  - 展示连接设备、品牌、型号、系统版本、应用版本、运行状态
- 用例执行
  - 选择设备、应用、测试包、执行范围，维护执行队列并触发任务
- 执行结果
  - 展示任务历史、状态、日志、HTML 报告入口
- 测试报告
  - 展示任务级统计与用例级明细，支持截图/视频预览

### 5.3 前端交互特征

- 首屏加载时拉取依赖状态、应用列表、设备列表
- 通过轮询刷新设备状态、任务状态、任务历史、报告数据
- 根据任务状态自动切换当前上下文
- 保持报告分页、状态筛选等交互能力

### 5.4 当前前端特点

- UI 功能集中在单文件，开发效率高，但组件边界较弱
- 前后端接口契约高度依赖当前 `App.tsx` 的数据结构
- 后续可逐步拆分为 `devices`、`runner`、`results`、`report` 四类子模块

## 6. 核心业务流程设计

### 6.1 系统启动流程

1. 加载环境变量
2. 判断运行模式并计算资源路径
3. 初始化服务容器
4. 初始化 SQLite 数据库
5. 按需启动远程 WebSocket
6. 启动 HTTP 服务
7. 自动打开前端页面

### 6.2 设备发现流程

1. 前端调用 `POST /api/list_devices`
2. 后端执行 `adb devices`
3. 并发读取设备属性和目标应用版本
4. 叠加数据库中的设备运行态
5. 返回给前端展示

### 6.3 测试执行流程

1. 前端提交 `device`、`app_key`、`suite`、`test_packages`
2. 后端校验参数与 Appium 是否可用
3. 根据 `suite` 生成 pytest `-m` 表达式
4. 校验设备是否已有任务运行
5. 创建任务 ID、日志文件、报告目录
6. 启动 pytest 子进程
7. 写入内存任务状态和 SQLite 历史记录
8. watcher 线程持续消费输出并等待结束
9. 生成 HTML 报告，复制最新报告副本
10. 解析 JSON 结果并写入报告表
11. 更新任务状态与设备状态
12. 前端轮询获取日志、历史与报告数据

### 6.4 报告展示流程

1. 前端选择任务
2. 调用 `GET /api/task_report_data/<task_id>`
3. 后端读取摘要、明细、分页结果
4. 为截图/视频生成 `/api/report_asset` 访问地址
5. 前端展示统计卡片、用例表格和附件预览

### 6.5 远程控制流程

1. 本地客户端根据环境变量建立 WebSocket 连接
2. 首次连接后发送 `register`
3. 定时发送 `heartbeat`
4. 远端下发 `command`
5. 本地路由到 `services_container.remote_ws_exec_command`
6. 返回 `response`

## 7. API 设计概览

### 7.1 设备相关

- `POST /api/list_devices`
- `GET /api/device_status/<device_serial>`

### 7.2 执行相关

- `GET /api/get_app_options`
- `POST /api/list_test_packages`
- `POST /api/run_tests`
- `POST /api/stop_task`
- `GET /api/task_status/<task_id>`
- `GET /api/task_history`
- `GET /api/task_log/<task_id>`

### 7.3 报告相关

- `GET /api/task_report/<task_id>`
- `GET /api/task_report_data/<task_id>`
- `GET /api/report_asset`
- `POST /api/open_report`

### 7.4 运行探针与远程相关

- `GET /api/startup_info`
- `GET /api/appium_ready`
- `GET /api/remote_ws_status`
- `GET /api/remote_ws_log`

设计原则：

- API 层只做参数解析和响应封装
- 真实业务逻辑放在 service 中
- 接口返回结构尽量稳定，避免前端轮询契约漂移

## 8. 路径与部署设计

### 8.1 运行模式

系统支持两种运行模式：

- 源码模式
- PyInstaller 打包模式

### 8.2 路径策略

核心路径包括：

- `RESOURCE_ROOT`
- `RUNTIME_ROOT`
- `reports/`
- `ui/`

设计要求：

- 报告、数据库、日志等运行时产物写入 `RUNTIME_ROOT`
- 前端构建产物从 `ui/` 提供静态访问
- 不得将路径逻辑写死为源码目录

### 8.3 运行产物

- `reports/runtime_state.db`
- `reports/task-logs/`
- `reports/task-reports/<task_id>/`
- `reports/test_results.json`
- `reports/test_report.html`
- `reports/allure-results/`
- `reports/allure-html/`
- `reports/remote-ws.log`

## 9. 安全性与稳定性设计

### 9.1 并发控制

- 使用进程内锁保护任务状态表
- 通过 `device_running_task` 保证单设备互斥
- SQLite 使用 WAL 模式提升并发读写稳定性

### 9.2 容错处理

- Appium 不可用时拒绝启动任务
- 任务日志与报告后处理异常都会写回任务状态
- 远程 WebSocket 连接支持重连与错误记录
- 设备信息探测设置超时，避免长时间阻塞

### 9.3 可观测性

- 任务执行日志写入独立文件
- 远程控制日志单独落盘
- 前端可查看任务历史与报告结果
- 数据库存储关键执行轨迹，便于排查

## 10. 当前方案优点与局限

### 10.1 优点

- 架构清晰，入口、接口、服务职责分明
- 本地测试执行闭环完整，具备设备、任务、日志、报告全链路能力
- 兼容源码运行与打包运行，适合桌面工具分发
- 数据库存储简单直接，便于维护和迁移
- 报告既能看 HTML，也能做结构化分析

### 10.2 局限

- 前端逻辑集中在 `App.tsx`，维护成本会随功能增长上升
- 任务运行态主要依赖进程内存，重启后实时上下文有限
- 当前调度能力以“单机 + 单设备互斥”为主，不是分布式执行架构
- iOS 仅预留扩展空间，尚未形成完整设备接入实现
- 远程控制能力可用，但缺少更强的认证、审计与权限模型

## 11. 后续演进建议

### 11.1 短期建议

- 拆分前端页面与状态管理，降低 `App.tsx` 复杂度
- 统一接口返回模型与错误码，减少前后端隐式约定
- 补充任务状态机定义，明确 `running/success/failed/stopped/idle` 语义
- 增加后端单元测试与接口回归测试

### 11.2 中期建议

- 引入任务队列和更明确的调度层
- 为报告增加趋势分析与历史聚合能力
- 完善远程控制的鉴权、租户隔离和操作审计
- 引入更稳定的前端数据请求封装和模块化组件结构

### 11.3 长期建议

- 支持 iOS 设备与平台抽象
- 演进为多终端、多设备、多节点的分布式执行平台
- 增加用例标签、优先级、失败重试和计划任务能力
- 支持与 CI/CD、缺陷系统、消息通知平台集成

## 12. 结论

`app-auto-test` 当前已经具备“桌面测试控制台 + 自动化执行引擎 + 报告中心”的完整雏形。其核心设计优势在于以轻量 Flask 服务为本地控制面，以服务容器组织后端能力，以 SQLite 和本地文件系统承载运行态与结果数据，在复杂度可控的前提下实现了较完整的自动化测试平台能力。

后续建设建议沿着“前端模块化、状态持久化、调度增强、平台扩展、远程治理”五个方向演进，在保持现有结构稳定的前提下逐步提升系统的可维护性、可扩展性与平台化能力。
